# 🏗️ 智能体分工图（KSTAR × 多智能体架构）

## 核心框架：KSTAR 学习循环

本系统以 **KSTAR 循环** 为核心流程框架，由多个专业化智能体在不同阶段协同工作。

```
      ┌──────────────────────────────────────────────────────┐
      │                     KSTAR 循环                         │
      │                                                      │
      │   K ──→ S ──→ T ──→ A ──→ R ──→ 检查达标?           │
      │   ↑                               │                  │
      │   └────────── 循环重置 ────────────┘                  │
      │                                                      │
      │   熔断条件: 正确率 ≥ 80% 或 循环次数 ≥ 2              │
      └──────────────────────────────────────────────────────┘
```

## 二、Agent 分工映射到 KSTAR

### K 阶段：诊断已知

| 智能体 | 职责 | 何时调用 | API消耗 |
|--------|------|---------|---------|
| **诊断评估Agent** | 分析前测结果，生成能力诊断报告 | 每次循环开始时 | 1次 |

**本地逻辑（不调用API）：** 从题库按知识点随机抽题、统计正确率

### S + T 阶段：现状→目标

| 智能体 | 职责 | 何时调用 | API消耗 |
|--------|------|---------|---------|
| **差距分析Agent** | 分析差距，确定当前原子级T | 拿到K之后 | 1次 |

**核心原则：** T必须是一个原子级知识点（如"FOB与CIF的区别"），不是大章节。

### A 阶段：行动

| 智能体 | 职责 | 何时调用 | API消耗 |
|--------|------|---------|---------|
| **资源生成Agent** | 生成精简讲义+练习题 | 确定T之后 | 1-2次 |
| **资源搜索Agent** | 搜索外部补充资源（可选） | 可选 | 1次 |

**人类行动：** 学生按学习材料自主学习（0 API消耗）

### R 阶段：反思

| 智能体 | 职责 | 何时调用 | API消耗 |
|--------|------|---------|---------|
| **反思评估Agent** | 分析后测结果，判断是否达标 | 学生学完回到系统后 | 1次 |

**本地逻辑（不调用API）：** 从题库抽后测题、统计正确率

### 辅助 Agent（全流程可用）

| 智能体 | 职责 |
|--------|------|
| **质量审核Agent** | 审核生成内容的准确性（可在A阶段后调用） |
| **格式转换Agent** | 将内容转换为不同格式（输出时调用） |

## 三、Agent 协作流程（单轮循环）

```
┌─────────────────────────────────────────────────────────────────┐
│                        单轮 KSTAR 循环                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│ ┌─ K 阶段 ───────────────────────────────────────────────────┐  │
│ │ 本地: 从题库抽5题 → 学生作答 → 统计正确率                    │  │
│ │ Agent: 诊断评估Agent → 输出诊断报告 + 薄弱点标记             │  │
│ │ 消耗: 0次API(出题) + 1次API(诊断)                            │  │
│ └──────────────────────────────────────────────────────────────┘  │
│                            ↓                                     │
│ ┌─ S+T 阶段 ──────────────────────────────────────────────────┐  │
│ │ Agent: 差距分析Agent → 确定当前原子级T                      │  │
│ │ 输出: "当前目标：掌握FOB与CIF的区别"                          │  │
│ │ 消耗: 1次API                                                │  │
│ └──────────────────────────────────────────────────────────────┘  │
│                            ↓                                     │
│ ┌─ A 阶段 ────────────────────────────────────────────────────┐  │
│ │ Agent: 资源生成Agent → 生成学习材料(半页讲义+3-5道练习)     │  │
│ │ 本地: 展示给学生 → 学生自主学习(无时间限制)                  │  │
│ │ 消耗: 1-2次API                                              │  │
│ └──────────────────────────────────────────────────────────────┘  │
│                            ↓                                     │
│ ┌─ R 阶段 ────────────────────────────────────────────────────┐  │
│ │ 本地: 从题库抽3题 → 学生作答 → 统计正确率                    │  │
│ │ Agent: 反思评估Agent → 判断是否达标                         │  │
│ │ 消耗: 0次API(出题) + 1次API(评估)                            │  │
│ └──────────────────────────────────────────────────────────────┘  │
│                            ↓                                     │
│ ┌─ 决策分支 ─────────────────────────────────────────────────┐   │
│ │ 达标 (≥80%) → 标记完成 → 进入下一个知识点                    │  │
│ │ 不达标 + 循环次数=1 → K = K(旧)+一句话更新 → 再来一轮        │  │
│ │ 不达标 + 循环次数≥2 → 跳过 → "建议求助人类老师"              │  │
│ └──────────────────────────────────────────────────────────────┘  │
│                                                                 │
│ 单轮最大消耗: 4-5次API调用                                      │
│ 每知识点封顶: 2轮 × 5次 = 10次API调用                           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 四、熔断与防破产机制

| 机制 | 规则 | 目的 |
|------|------|------|
| 达标阈值 | 正确率 ≥ 80% 视为通过 | 不需要100%完美，避免无限循环 |
| 最大循环 | 每个知识点最多循环2次 | 防止卡死在一个点上 |
| 求助兜底 | 2次不达标 → 建议咨询人类老师 | 承认AI的局限，避免虚假的"教会了" |
| K更新成本 | 每次只用一句话更新状态 | 控制在1次API调用内 |
| 题库兜底 | 某知识点题库不足3题→触发大模型补充生成 | 不依赖题库完整度 |

## 五、本地题库设计

```
题库结构（SQLite / JSON）:

知识点表:
  - id, name, course, difficulty (基础/进阶/挑战)

题目表:
  - id, knowledge_point_id, question_text, 
    options (JSON), correct_answer, 
    type (单选/多选/判断), difficulty

学生状态表:
  - student_id, knowledge_point_id, 
    status (未学/学习中/已掌握/已跳过),
    correct_count, total_count, 
    history (答题记录JSON)
```

**运行时的出题策略：**
1. 优先从本地题库随机抽取
2. 若该知识点题库题量 < 3 道 → 触发大模型生成补充题
3. 补充题自动入库，后续复用
